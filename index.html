CRITICAL CASE-001  ·  2025-XX-XX 14:30 UTC
Operation Web Shell — Threat Hunt: www.company.com
TRUE POSITIVE

As a Tier 1 Analyst on a routine threat hunt, the web team reported the main company website (www.company.com / 10.0.0.5) felt slow — but no automated alerts had fired. With no SIEM hit to guide us, I pulled one hour of raw firewall and web server logs and began a manual correlation analysis. The investigation uncovered a full attacker lifecycle: reconnaissance → exploitation → web shell deployment → command execution.

// ATTACK TIMELINE RECONSTRUCTION

14:30:01 – 14:30:03
Attacker (88.198.5.10) attempts to connect to ports 22 (SSH), 3389 (RDP), and 445 (SMB). All three are DENIED by the firewall — standard lateral movement / initial access probing.
14:30:04
Port 80 (HTTP) is open. Attacker pivots to web attack surface. First GET to /index.html — 200 OK — confirms server is live.
14:30:05 – 14:30:07
Attacker uses curl/7.68 to probe for sensitive files: /admin.php, /config.txt, /backup.zip. All return 404 — active reconnaissance / directory brute-forcing in progress.
14:30:08
GET request to /upload.php returns 200 OK. A file upload endpoint has been discovered — this is the attack vector.
14:30:09 — EXPLOITATION
POST request to /upload.php — the attacker uploads a malicious PHP web shell file (shell.php) to the server via the unsecured upload endpoint.
14:30:10 — COMMAND EXECUTION
GET to /uploads/shell.php?cmd=whoami returns 200. The web shell is live and executing OS commands. The attacker is testing their access level with whoami. Server is fully compromised.

// DETAILED QUESTION ANALYSIS

[ 1 ] FIREWALL ANALYSIS
What is the attacker's IP and the server's IP?
Attacker IP: 88.198.5.10 — identified by three rapid DENY hits on non-HTTP ports followed by sustained HTTP traffic to the web server. Server IP: 10.0.0.5 (www.company.com).
What 3 ports did the attacker try to connect to that were blocked?
Ports 22 (SSH), 3389 (RDP), and 445 (SMB) — all DENIED. This is characteristic of a port scan targeting high-value remote access and file-sharing services before pivoting to the web application.
What is one port that was open?
Port 80 (HTTP) — ALLOWed. The attacker pivoted entirely to this vector after the other ports were blocked.
[ 2 ] WEB SERVER ANALYSIS
What 3 filenames did the attacker look for that did not exist?
/admin.php, /config.txt, and /backup.zip — all returning 404 Not Found. These are common targets in web directory enumeration attacks (sensitive admin panels, configuration files, database backups).
What filename did they find that did exist?
/upload.php — returned 200 OK. This unsecured file upload endpoint became the primary attack vector for deploying the web shell.
What user_agent is the attacker using? Is it a normal browser?
curl/7.68 — this is a command-line HTTP tool, not a browser. No legitimate end user browses a website using curl. This strongly indicates automated/scripted attack activity, and combined with the 404 enumeration pattern, is a high-confidence indicator of malicious behaviour.
[ 3 ] THE BREACH — LOG CORRELATION
At 14:30:09, the attacker makes a POST request. What were they doing?
The POST to /upload.php is the web shell being uploaded. Having confirmed the endpoint is live (14:30:08 GET returned 200), the attacker used curl to POST a malicious PHP file — shell.php — to the server. The server accepted it (200 OK), placing a remote code execution backdoor on the filesystem.
At 14:30:10, the "smoking gun" appears. What is the full URI?
Full URI: /uploads/shell.php?cmd=whoami — The shell was deposited in the /uploads/ directory (a common misconfiguration where uploaded files are web-accessible and executable). The attacker immediately queried it with a test command via the cmd parameter.
What specific command did the attacker run?
whoami — a standard privilege discovery command. The attacker is checking what user account the web server process is running as (e.g., www-data, root, etc.). This determines whether privilege escalation is needed before moving further.

Two distinct MITRE ATT&CK techniques are identifiable in this attack chain, mapped below with supporting evidence from the logs.

// Identified Tactics & Techniques
TA0043Reconnaissance T1595Active Scanning

The stream of 404 errors from GET requests to /admin.php, /config.txt, and /backup.zip represents active web directory enumeration — the attacker is systematically probing for sensitive or exploitable resources on the target server.

This is consistent with automated directory brute-forcing tools (common ones include gobuster, dirbuster, ffuf). The use of curl/7.68 as the user-agent confirms scripted, non-human browsing behaviour. The attacker was probing for vulnerabilities and entry points before committing to an exploitation path.

TA0003Persistence T1505.003Web Shell TA0011Command & Control

The deployment of /uploads/shell.php is a textbook Web Shell (T1505.003). A web shell is a malicious script placed on a compromised web server that allows an attacker to execute arbitrary OS commands remotely through HTTP requests — effectively turning the web server into a C2 node.

The ?cmd=whoami query parameter is the classic interface of a PHP web shell. This technique is dangerous because it blends into legitimate web traffic on port 80/443, often bypassing perimeter firewall rules. It also establishes persistence — the shell remains on the server until manually removed.

Attack Phase 1
Reconnaissance (T1595)
Attack Phase 2
Initial Access via Upload (T1190)
Attack Phase 3
Web Shell Exec (T1505.003)
⚠ VERDICT: TRUE POSITIVE — ACTIVE COMPROMISE

This is unambiguously a True Positive. The evidence chain is complete: port scan → web directory enumeration → upload endpoint discovery → web shell upload → OS command execution. There is no plausible benign explanation for a curl client posting files to an upload endpoint and immediately querying a PHP file with OS commands. Immediate containment and escalation are required.

  • ISOLATE
    Isolate the Web Server (10.0.0.5) Immediately remove 10.0.0.5 from the network. The web shell is live — every second it remains connected is a second the attacker can escalate privileges, exfiltrate data, or pivot to internal systems. Do not shut the server down; isolate it to preserve volatile memory and log forensic evidence.
  • ESCALATE
    Escalate to the Incident Response Team (IRT) This incident exceeds Tier 1 scope. Escalate immediately with a full timeline, IOCs (attacker IP, shell path, commands run), and log evidence. The IRT will lead forensic investigation, determine full blast radius, and manage remediation.
  • BLOCK
    Block Attacker IP at Firewall — 88.198.5.10 Create a firewall rule to block all inbound traffic originating from 88.198.5.10 across all ports. This prevents the attacker from reaching any other internal resources. Also consider adding this IP to a threat intelligence blocklist and cross-checking it against other log sources to identify any other systems it may have touched.
  • PRESERVE
    Preserve Evidence — Do Not Wipe Before any remediation, preserve disk images, memory dumps, and all relevant log files. The whoami command suggests the attacker may still be in the reconnaissance phase — a forensic review of what they accessed is critical to assessing the full scope of the breach.
HIGH CASE-002  ·  Network Forensics
Suspicious Outbound Traffic — Wireshark PCAP Analysis
TRUE POSITIVE

An alert from the network monitoring team flagged anomalous outbound DNS queries from an internal workstation (192.168.1.45). A packet capture was taken and loaded into Wireshark for analysis. The volume and structure of DNS queries suggested data exfiltration via DNS tunnelling.

Source Host
192.168.1.45
Protocol Abused
DNS (Port 53)
Suspicious Domain
exfil.attacker-c2.net
Query Volume
847 queries / 10 min

// WIRESHARK DISPLAY FILTER APPLIED: dns && ip.src == 192.168.1.45

Frame 1024: DNS Query YWRtaW5AY29tcGFueS5jb20K.exfil.attacker-c2.net (Type A) Frame 1025: DNS Query cGFzc3dvcmQxMjM0NQo=.exfil.attacker-c2.net (Type A) Frame 1026: DNS Query c2VjcmV0a2V5X2FwaQ==.exfil.attacker-c2.net (Type A) Frame 1027: DNS Response: NXDOMAIN ← domain does not resolve ... [+847 similar queries in 10 minutes, all to *.exfil.attacker-c2.net]
What made this traffic suspicious in Wireshark?
Normal DNS query volumes are low (tens per minute). 847 DNS queries in 10 minutes to a single external domain is a massive anomaly. The subdomains contained Base64-encoded strings — a hallmark of DNS tunnelling, where data is encoded into DNS hostnames to bypass firewalls that only block known-bad IPs.
What do the Base64 subdomains decode to?
Decoded: admin@company.com, password12345, secret_key_api — credential and API key data being exfiltrated chunk-by-chunk via DNS queries. Each query carries a small payload encoded in the subdomain label, reconstructed by the attacker's C2 server.
How was Wireshark used to confirm this?
Wireshark's Statistics → DNS panel revealed the query rate spike. The display filter dns.qry.name contains "attacker-c2.net" isolated all suspicious queries. Following the UDP stream showed the sequential nature of the queries, consistent with automated tunnelling software rather than organic browsing.
TA0010Exfiltration T1048.003Exfil Over DNS

Data exfiltration over DNS (T1048.003) — encoding sensitive data into DNS query hostnames and sending them to an attacker-controlled DNS server that logs the subdomains. DNS traffic is commonly allowed outbound through firewalls, making it a stealthy exfiltration channel.

TA0011Command & Control T1071.004DNS C2

The attacker's C2 infrastructure operates over DNS, communicating with the implant on the compromised host via DNS query/response cycles — a technique that evades many traditional network security controls focused on TCP/HTTP.

MEDIUM CASE-003  ·  SIEM Investigation
Credential Brute-Force — Splunk SIEM Detection
TRUE POSITIVE

A Splunk alert triggered for multiple failed authentication events originating from a single external IP within a 5-minute window. Investigation revealed a systematic credential brute-force attack targeting the company's VPN gateway. Cross-referencing with Active Directory logs confirmed the targeted account and a subsequent successful login.

Attack Source
45.227.18.103
Target
VPN Gateway / AD
Failed Attempts
312 in 5 minutes
Successful Login
YES — j.smith

// SPLUNK SPL — FAILED LOGIN SPIKE DETECTION

index=windows_security EventCode=4625 | stats count AS failed_attempts BY src_ip, TargetUserName | where failed_attempts > 50 | sort - failed_attempts -- RESULTS -- 45.227.18.103 | j.smith | 312 failed attempts 45.227.18.103 | admin | 87 failed attempts 45.227.18.103 | svc_acc | 54 failed attempts

// CROSS-REFERENCE: SUCCESSFUL LOGINS FOLLOWING FAILURES (EventCode=4624)

index=windows_security EventCode=4624 src_ip=45.227.18.103 | table _time, src_ip, TargetUserName, LogonType, WorkstationName -- RESULTS -- 15:47:23 | 45.227.18.103 | j.smith | LogonType=3 (Network) | ← SUCCESSFUL BRUTE FORCE
What triggered the initial Splunk alert?
The alert rule was configured to fire when a single source IP generates more than 50 EventCode 4625 (Windows Failed Logon) events in a 5-minute window. 45.227.18.103 generated 312 failures across multiple accounts — well above the threshold, indicating systematic credential stuffing or dictionary attack activity.
Why is the EventCode 4624 finding so critical?
A successful login (EventCode 4624) from the same attacking IP, targeting account j.smith — the most heavily targeted account — after 312 failures confirms the brute-force was successful. The attacker now has authenticated access to the network. This elevates the severity and requires immediate account lockout and password reset as a minimum response.
How does Splunk help in this investigation?
Splunk's SPL (Search Processing Language) enabled fast aggregation across millions of log events. The stats count by src_ip query surfaced the attack pattern in seconds — work that would take hours manually. Correlating the 4625 and 4624 events from the same source IP is the kind of multi-source correlation that makes SIEM indispensable in a SOC.
TA0006Credential Access T1110.001Brute Force: Password Guessing TA0001Initial Access

T1110.001 — Brute Force credential attack against a network-accessible authentication portal (VPN). The attacker systematically tried a large number of username/password combinations until one succeeded. The successful authentication (LogonType 3) then provides Initial Access to the internal network.
